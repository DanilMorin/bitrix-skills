# Повторяющиеся блоки и будущие Bitrix-компоненты

## 1. Зачем искать повторяющиеся блоки

В статической вёрстке часто один и тот же HTML-каркас копируется много раз:

* карточки статей;
* карточки кейсов;
* карточки услуг;
* отзывы;
* этапы работы;
* преимущества;
* FAQ;
* элементы слайдера;
* элементы меню;
* теги;
* похожие материалы.

При посадке на Bitrix такие блоки обычно становятся шаблонами компонентов или частью шаблонов компонентов.

Если заранее не найти повторения, при интеграции появятся проблемы:

* дублирование HTML в нескольких файлах;
* сложная поддержка;
* разные классы для одинаковой структуры;
* трудности при подключении инфоблоков;
* несогласованное отображение карточек;
* риск сломать адаптивность.

## 2. Что считать повторяющимся блоком

Блок считается повторяющимся, если у него:

* одинаковая или почти одинаковая HTML-структура;
* одинаковая роль на странице;
* меняется только текст, изображение, ссылка, дата, теги или порядок элементов;
* одинаковые или похожие CSS-классы;
* одинаковое поведение в JavaScript.

Пример:

```html
<div class="cases-item">
  <img class="cases-item__bg" src="...">
  <div class="cases-item__content">
    <div class="cases-item__tags">...</div>
    <p class="cases-item__desc">...</p>
  </div>
</div>
```

Если такой блок повторяется много раз, он является кандидатом на будущий список элементов.

## 3. Что меняется внутри повторяемого блока

При анализе нужно выписать, какие данные меняются:

* заголовок;
* описание;
* изображение;
* ссылка;
* дата;
* теги;
* иконка;
* подпись;
* цена;
* кнопка;
* порядок элементов;
* модификатор внешнего вида.

Эти данные в будущем могут стать:

* полями элемента инфоблока;
* свойствами инфоблока;
* пользовательскими полями;
* параметрами компонента;
* настройками шаблона компонента.

## 4. Типы повторяющихся блоков

### Карточка списка

Будущий вариант:

```text
bitrix:news.list
```

Примеры:

* статьи;
* кейсы;
* услуги;
* новости;
* похожие материалы;
* сотрудники;
* отзывы.

### Детальная страница

Будущий вариант:

```text
bitrix:news.detail
```

Примеры:

* страница статьи;
* страница кейса;
* страница услуги;
* страница новости.

### Комплексный раздел

Будущий вариант:

```text
bitrix:news
```

Примеры:

* блог со списком и детальными страницами;
* кейсы со списком и детальными страницами;
* услуги со списком и детальными страницами.

### Меню

Будущий вариант:

```text
bitrix:menu
```

Примеры:

* главное меню;
* футер-меню;
* мобильное меню;
* меню раздела.

### Хлебные крошки

Будущий вариант:

```text
bitrix:breadcrumb
```

### Статичный редактируемый блок

Будущий вариант:

```text
включаемая область
```

Примеры:

* CTA;
* небольшой текстовый блок;
* контакты;
* подпись;
* блок с реквизитами;
* простая промо-секция.

### Сложный интерактивный блок

Будущий вариант:

```text
кастомный компонент
```

Примеры:

* сложный калькулятор;
* интерактивная карта;
* сложный фильтр;
* нестандартная форма;
* блок с ajax-логикой.

## 5. Как выбирать: общий компонент или разные компоненты

Если HTML одинаковый, а меняются только данные — нужен один общий шаблон.

Если HTML почти одинаковый, но отличаются визуальные состояния — можно использовать общий блок и модификаторы.

Пример:

```html
<article class="content-card content-card--case"></article>
<article class="content-card content-card--blog"></article>
```

Если похожа только идея, но структура и поведение разные — лучше оставить разные блоки, но описать их связь в карте компонентов.

## 6. Что нельзя делать автоматически

AI-ассистент не должен:

* объединять блоки только по похожему названию;
* менять классы без оценки риска;
* удалять визуальные отличия;
* превращать всё подряд в компоненты;
* переписывать HTML, если достаточно описать будущую архитектуру;
* ломать текущий статический preview.

## 7. Формат отчёта по повторяющимся блокам

Использовать таблицу:

| Блок | Где найден | Повторяется | Меняется | Будущая Bitrix-сущность | Что сделать сейчас | Риск |
| ---- | ---------- | ----------- | -------- | ----------------------- | ------------------ | ---- |

## 8. Уровни риска

Low:

* вынести inline-style в SCSS;
* добавить alt;
* добавить data-атрибут;
* улучшить нейминг без удаления старого класса;
* сгруппировать SCSS без изменения визуала.

Medium:

* объединить похожие блоки;
* добавить общий класс;
* изменить структуру SCSS;
* перенести JS в отдельный модуль.

High:

* переименовать основные классы;
* изменить HTML-структуру карточек;
* заменить статику на динамику;
* переводить HTML в PHP;
* менять сборку проекта.
