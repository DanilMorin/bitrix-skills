# Основы интеграции вёрстки с 1С-Битрикс

Источник: официальная документация Bitrix Framework, раздел «Особенности интеграции с Битрикс».

## 1. Главный принцип

При подготовке вёрстки к 1С-Битрикс нужно заранее разделять проект на две большие части:

1. Вёрстка шаблона сайта.
2. Вёрстка компонентов.

Это ключевой принцип, который влияет на структуру HTML, SCSS, JavaScript и будущую архитектуру проекта.

## 2. Шаблон сайта

Шаблон сайта отвечает за общий внешний вид сайта.

К шаблону сайта относятся:

* header;
* footer;
* общая сетка страницы;
* глобальные контейнеры;
* общая типографика;
* глобальные стили;
* общие декоративные элементы;
* layout страницы;
* колонки;
* общие зоны, которые повторяются на большинстве страниц.

В будущем эти элементы обычно переносятся в структуру:

```text
/local/templates/site_name/
  header.php
  footer.php
  template_styles.css
  components/
```

На этапе статической вёрстки не нужно сразу создавать эти PHP-файлы, но нужно писать HTML/SCSS так, чтобы разделение на header, footer и рабочую область было очевидным.

## 3. Компоненты

Компонент — это логически независимая часть страницы.

К компонентам относятся блоки, которые:

* имеют повторяющуюся HTML-структуру;
* выводят список элементов;
* имеют детальную страницу;
* могут переиспользоваться на разных страницах;
* имеют свои данные;
* могут иметь собственный JavaScript;
* могут иметь собственные стили.

Примеры будущих компонентов:

* список статей;
* список кейсов;
* карточки услуг;
* похожие статьи;
* отзывы;
* FAQ;
* форма обратной связи;
* меню;
* хлебные крошки;
* слайдер;
* фильтр.

На этапе статической вёрстки важно не делать такие блоки хаотичными. Их структура должна быть предсказуемой и переносимой.

## 4. Почему это важно

Если вёрстка заранее не разделена на шаблон сайта и компоненты, при посадке на Bitrix появляются проблемы:

* сложно понять, что должно быть в header.php;
* сложно понять, что должно быть в footer.php;
* повторяющиеся карточки приходится вручную переносить в шаблоны компонентов;
* стили начинают конфликтовать;
* JavaScript ломается после динамического вывода данных;
* одинаковые блоки приходится поддерживать в нескольких местах;
* сложнее подключать инфоблоки;
* сложнее сделать редактирование из админки.

## 5. Контекст использования

Перед подготовкой вёрстки нужно понимать, где она будет использоваться:

* в обычном сайте на «1С-Битрикс: Управление сайтом»;
* в корпоративном портале;
* в интернет-магазине;
* в готовом решении;
* в публичной части с административной панелью;
* в мобильном интерфейсе.

Для обычной посадки корпоративного сайта чаще всего важен контекст «Управление сайтом».

В этом контексте компоненты могут использоваться в разных дизайнах, поэтому их вёрстка должна быть устойчивой, понятной и не слишком зависимой от внешнего окружения.

## 6. Устойчивость к внешним стилям

Bitrix может подключать свои стили, стили шаблона, стили компонентов и дополнительные библиотеки.

Поэтому при подготовке вёрстки важно:

* не использовать слишком общие классы;
* не писать стили на голые теги без контекста;
* не завязывать компонент на родителя, который может исчезнуть;
* избегать чрезмерной вложенности;
* не полагаться на случайный порядок подключения CSS;
* делать классы блоков достаточно уникальными;
* не смешивать стили разных компонентов.

Плохо:

```scss
.card {
  ...
}

.title {
  ...
}

.content {
  ...
}
```

Лучше:

```scss
.case-card {
  ...
}

.article-card {
  ...
}

.project-hero {
  ...
}
```

Или через БЭМ:

```scss
.cases-item {
  &__title {}
  &__image {}
  &__tags {}
}
```

## 7. Bootstrap и Bitrix

В Bitrix Framework может использоваться Bootstrap. В документации отмечается, что Bootstrap подключается не в конкретном компоненте, а в целом на продукте из модуля main.

Из-за этого при вёрстке нужно учитывать возможные конфликты с классами Bootstrap.

Не рекомендуется без необходимости использовать классы, которые могут конфликтовать с Bootstrap:

* container;
* row;
* col;
* card;
* btn;
* modal;
* nav;
* active;
* dropdown.

Если такие классы уже есть в проекте, нужно оценить риск. Не всегда их нужно сразу переименовывать, потому что это может сломать вёрстку. Но в отчёте нужно отметить потенциальный конфликт.

## 8. Где должны лежать стили

В Bitrix есть разница между стилями шаблона сайта и стилями компонента.

Будущая логика:

```text
template_styles.css — стили общего шаблона сайта;
style.css внутри шаблона компонента — стили конкретного компонента.
```

На этапе статической SCSS-вёрстки можно готовить проект так:

```text
src/scss/
  base/
  layout/
  components/
  pages/
```

Где:

* `base` — переменные, reset, типографика;
* `layout` — header, footer, container, grid;
* `components` — карточки, кнопки, формы, слайдеры, теги;
* `pages` — стили конкретных страниц.

## 9. Что проверять в статической вёрстке

Перед будущей посадкой на Bitrix нужно проверить:

* где заканчивается header;
* где начинается рабочая область страницы;
* где начинается footer;
* какие блоки являются повторяемыми;
* какие блоки должны редактироваться из админки;
* какие данные должны стать свойствами инфоблока;
* какие списки станут `news.list`;
* какие страницы станут `news.detail`;
* какие блоки лучше оставить статичными;
* какие блоки вынести во включаемые области;
* какие JS-скрипты завязаны на статичный HTML.

## 10. Правило для AI-аудита

При анализе проекта AI-ассистент не должен сразу переписывать проект под Bitrix.

Сначала он должен составить карту:

1. Шаблон сайта.
2. Компоненты.
3. Включаемые области.
4. Инфоблоки.
5. Статичные блоки.
6. Проблемные места.
7. Безопасные изменения.
8. Изменения, которые лучше отложить до посадки.

Только после этого можно приступать к безопасному рефакторингу.
