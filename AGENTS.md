# Bitrix Skill — правила для AI-ассистента

Этот репозиторий содержит правила, чеклисты, шаблоны отчётов и промпты для анализа статической HTML/SCSS/JS-вёрстки перед будущей посадкой на 1С-Битрикс.

## Главная цель

Помогать анализировать и улучшать статическую вёрстку так, чтобы в будущем её было проще перенести на 1С-Битрикс.

Важно: skill не требует сразу переводить проект на PHP и не требует создавать полноценный Bitrix-шаблон на этапе аудита.

## Основной сценарий

Проект может быть статической HTML/SCSS/JS/Gulp-вёрсткой, которая сейчас показывается заказчику через GitHub Pages или другой статический preview.

AI-ассистент должен:

1. Сначала понять структуру проекта.
2. Найти повторяющиеся блоки.
3. Определить, какие блоки в будущем станут Bitrix-компонентами.
4. Определить, какие блоки относятся к шаблону сайта.
5. Найти проблемы, которые усложнят будущую посадку на Bitrix.
6. Предложить безопасные изменения, которые не ломают текущую вёрстку.
7. Не переносить проект на PHP без отдельной команды пользователя.

## Базовый принцип Bitrix

При анализе учитывать разделение вёрстки на:

1. Вёрстку шаблона сайта.
2. Вёрстку компонентов.

Шаблон сайта — это общий layout проекта: header, footer, общие обёртки, сетки, колонки, глобальные стили, типографика, общие элементы.

Компонент — это логически независимая часть страницы, у которой могут быть свои данные, HTML-шаблон, стили и JavaScript. Компонент должен быть переносимым и переиспользуемым.

## Что нельзя делать без прямого разрешения

* Не переводить HTML-файлы в PHP.
* Не создавать Bitrix-проект внутри статической вёрстки.
* Не заменять статический HTML на `IncludeComponent` без отдельного задания.
* Не ломать GitHub Pages preview.
* Не менять названия классов без причины.
* Не переписывать большие файлы целиком, если можно исправить локально.
* Не удалять визуальные различия между блоками только потому, что у них похожая структура.
* Не добавлять тяжёлые зависимости без необходимости.
* Не менять сборку проекта без отдельного согласования.

## Что нужно искать при аудите

* Повторяющиеся HTML-структуры.
* Карточки списков.
* Детальные страницы.
* Статичные блоки, которые можно вынести во включаемые области.
* Глобальные элементы шаблона сайта.
* Inline-style.
* Хардкод ссылок.
* Хардкод повторяющихся данных.
* JS, завязанный на случайный порядок элементов.
* Стили, которые могут конфликтовать с Bitrix или Bootstrap.
* Слишком общие классы.
* Слишком глубокую SCSS-вложенность.
* Неустойчивую адаптивность.

## Типы будущих Bitrix-сущностей

При анализе относить блоки к одному из типов:

1. Site template — header, footer, layout, глобальные элементы.
2. Include area — редактируемый статичный блок.
3. Menu — навигация.
4. Breadcrumbs — цепочка навигации.
5. News list — список статей, кейсов, услуг, новостей.
6. News detail — детальная страница статьи, кейса, услуги.
7. Catalog section — список элементов каталожного типа.
8. Form component — форма обратной связи.
9. Custom component — сложный блок с нестандартной логикой.
10. Static block — блок, который пока не нужно выносить в Bitrix.

## Уровни риска

Каждое предложение должно иметь уровень риска:

* low — можно делать сейчас, GitHub Pages и визуал почти точно не сломаются.
* medium — можно делать, но нужна визуальная проверка.
* high — лучше отложить до этапа посадки на Bitrix.

## Проверки

Если в проекте есть команды проверки, перед финальным ответом предложить или выполнить:

* npm run check
* npm run lint
* npm run build

Если команды неизвестны, сначала посмотреть package.json.
